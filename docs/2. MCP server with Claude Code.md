# Managing MCP Servers in Claude Code

before we start


When Claude Code spawns the MCP server as a child process, fnm's shell shims may not be available, so `node` could fail to resolve.

The safest approach is to get the full path upfront:

```bash
which node
```

This will give you something like:

```
/home/artyom/.local/share/fnm/node-versions/v22.11.0/installation/bin/node
```

Use that full path in your config:

```json
{
  "mcpServers": {
    "my-server": {
      "command": "/home/artyom/.local/share/fnm/node-versions/v22.11.0/installation/bin/node",
      "args": ["./my-mcp/main.js"]
    }
  }
}
```

The downside is this path is tied to a specific Node version, so if you upgrade Node via fnm later and switch versions, you'd need to update the path in all your MCP configs.

A cleaner workaround is to create a small wrapper script:

```bash
#!/bin/bash
# ~/bin/node-fnm
export PATH="/home/artyom/.local/share/fnm/node-versions/v22.11.0/installation/bin:$PATH"
exec node "$@"
```

Make it executable:
```bash
chmod +x ~/bin/node-fnm
```

Then use that in your config:
```json
{
  "command": "/home/artyom/bin/node-fnm"
}
```

But honestly the simplest thing is just use `which node` each time you switch Node versions and update accordingly. Most people don't switch versions that often in practice.

## How MCP Servers Work

MCP servers in Claude Code are **not persistent background services**. Claude Code launches them automatically as child processes when a session starts, and shuts them down when the session ends. You don't manually start or stop them — Claude Code manages their lifecycle.

---

## Starting MCP Servers

MCP servers start automatically when you begin a Claude Code session, as long as they are registered in your config. There is no manual start command.

If you want to verify your servers are registered:

```bash
claude mcp list
```

To start a session and have your servers available:

```bash
claude  # starts Claude Code, all registered MCP servers launch automatically
```

---

## Stopping MCP Servers

MCP servers stop automatically when your Claude Code session ends (e.g. you type `/exit` or close the terminal). Again, no manual stop command needed.

---

## Adding and Removing Servers (the real "management")

Since start/stop is automatic, the real management is about registering and unregistering servers.

### Add a server

```bash
# Local stdio server (most common for workshops)
claude mcp add <name> <command> [args...]

# Example: add a local Node.js MCP server
claude mcp add weather-server node /home/user/mcp/weather.js

# Add globally (available in all projects)
claude mcp add --scope user weather-server node /home/user/mcp/weather.js

# Add a remote HTTP server
claude mcp add --transport http <name> <url>

# Example: add a remote HTTP server
claude mcp add --transport http my-api https://api.example.com/mcp

# Add with environment variables
claude mcp add weather-server node /home/user/mcp/weather.js \
  --env API_KEY=your-key-here \
  --env NODE_OPTIONS=--no-deprecation
```

### Remove a server

```bash
# Remove from current project scope
claude mcp remove <name>

# Remove from global scope
claude mcp remove --scope user <name>
```

### List all registered servers

```bash
claude mcp list
```

### Get details about a specific server

```bash
claude mcp get <name>
```

## Managing Servers via Config Files

Instead of using CLI commands, you can edit config files directly.

### Global config (all projects)

File location: `~/.claude.json`

```json
{
  "mcpServers": {
    "weather-server": {
      "type": "stdio",
      "command": "node",
      "args": ["/home/user/mcp/weather.js"],
      "env": {
        "NODE_OPTIONS": "--no-deprecation"
      }
    },
    "my-api": {
      "type": "http",
      "url": "https://api.example.com/mcp"
    }
  }
}
```

### Per-project config (shared with team via git)

File location: `.mcp.json` in your project root

```json
{
  "mcpServers": {
    "issues-server": {
      "type": "stdio",
      "command": "node",
      "args": ["./my-mcp/main.js"],
      "env": {
        "NODE_OPTIONS": "--no-deprecation",
        "API_BASE_URL": "http://localhost:3000/api"
      }
    }
  }
}
```

> `.mcp.json` is meant to be committed to git so your whole team shares the same MCP server definitions.

## Disabling a Server Temporarily

There is no "pause" command. Your options are:

**Option 1 — Remove and re-add later:**
```bash
claude mcp remove weather-server
# later...
claude mcp add weather-server node /home/user/mcp/weather.js
```

**Option 2 — Comment it out in the config file** (`~/.claude.json` or `.mcp.json`):
```json
{
  "mcpServers": {
    // "weather-server": { ... }   <-- commented out = disabled
    "issues-server": { ... }
  }
}
```

> Note: JSON doesn't officially support comments, but Claude Code's config parser accepts them. If another tool parses the file it may break.

**Option 3 — Rename the key** in the config so Claude Code doesn't recognize it:
```json
{
  "mcpServers": {
    "_disabled_weather-server": { ... },
    "issues-server": { ... }
  }
}
```

## Scopes Summary

| Scope                | File                           | When to use                        |
| -------------------- | ------------------------------ | ---------------------------------- |
| **user** (global)    | `~/.claude.json`               | Servers you want in every project  |
| **project** (shared) | `.mcp.json` in project root    | Team-shared servers, commit to git |
| **local** (default)  | `~/.claude.json` local section | Personal overrides for one project |

> Avoid defining the same server in multiple scopes — it causes duplicates and unexpected behavior.

## Troubleshooting

If a server is registered but not working after starting a session:

```bash
# Check what's registered
claude mcp list

# Check a specific server's config
claude mcp get <name>

# Test your MCP server manually via stdio
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' | node /path/to/server.js
```

Common issues:
- Wrong path to the server file
- Node.js not found — use the full path from `which node`
- Server throws an error on startup — test it manually via stdio first
- Same server defined in multiple scopes — remove the duplicate
